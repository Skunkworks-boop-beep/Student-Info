PROJECT OVERVIEW  1.1 Purpose & Scope  Student.Info is a full-stack web/mobile application developed by the BAU University MIS Club. It gives students a structured, transparent channel to submit complaints, track resolution progress, and engage with peer solutions — while providing administrators with analytics, workflow automation, and moderation tools. The system also integrates AI assistance, gamification, and multilingual support to maximise student engagement and satisfaction.  1.2 Team & Roles  Member  Role  Responsibilities Damilola  Tech Lead  Architecture, security oversight, code review, documentation Aseel  Frontend Dev  Student portal UI, component library, responsive design Erik  Backend Dev  REST API design, database management, workflow engine Abilio  Backend Dev  AI integrations, notification service, gamification logic Zanela  Full-Stack Dev  Admin portal, analytics dashboard, heat-map rendering Nora  UI/UX & QA  Design system, user testing, accessibility, test cases    1.3 Recommended Tech Stack Layer  Technology Frontend  React.js (Next.js), Tailwind CSS, Chart.js / Recharts, Leaflet.js Backend  Node.js + Express.js, Socket.IO (real-time notifications) Database  PostgreSQL (primary), Redis (sessions & caching) AI Layer  Anthropic Claude API (assistant, translation, content moderation) Auth  JWT + Refresh tokens, bcrypt (cost ≥12), optional Google SSO Storage  AWS S3 / Cloudinary (file attachments) DevOps  Docker, GitHub Actions CI/CD, Nginx reverse proxy, SSL/TLS Monitoring  Sentry (errors), Prometheus + Grafana (metrics), Uptime Robot    BAU University — Student.Info | Project Documentation v1.0 Page 4  2. SYSTEM ARCHITECTURE  2.1 High-Level Architecture  Student.Info follows a three-tier architecture: a React/Next.js presentation layer, a Node.js/Express API layer, and a PostgreSQL persistence layer. RECOMMENDATIONS & NEXT STEPS  Bus Tracker Integration  Partner with the university transport team to expose a real-time bus location API. Display ETA on the student app home screen. High-impact, low-effort once the data feed is available.  Complaint Category Taxonomy  Define a fixed taxonomy: Academic · Administrative · Facilities · IT · Transport · Safety · Other. Categories drive AI suggestions, video matching, routing rules, and heat-map filtering.  Appreciation Box  Alongside complaints, allow students to submit positive feedback and appreciation for staff. Separate feed in admin portal. Boosts morale and gives a balanced picture of service quality.  Evaluation / Rating System  After resolution, student rates the experience 1–5 stars with optional comment. Admin sees aggregate score per category and per team member. Essential for continuous improvement.  Offline Support (PWA)  Service worker allows drafting complaints offline. Sync when connection resumes. Critical for students in low-connectivity campus areas.  Multilingual Admin Portal  Admin portal currently assumed English-only. Add i18n for at least Arabic, French, and Turkish given BAU's international student body.  Data Anonymisation for Analytics  Apply k-anonymity (minimum group size of 5) before serving heat-map or category data to prevent reverse-identification of individuals.  Open Source Consideration  Consider open-sourcing the codebase under MIT licence. Increases trust, enables peer review, and may attract external contributors for future features.  Phased Rollout Plan  Pilot with one faculty before campus-wide launch. Collect feedback, iterate on UX, and stress-test notifications at scale. Define success KPIs before each rollout phase.
